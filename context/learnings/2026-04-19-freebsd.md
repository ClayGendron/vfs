# FreeBSD — Influences for VFS and FSP

> Reference repo: `/Users/claygendron/Git/Repos/freebsd-src`

## Overview

FreeBSD's VFS (Virtual File System) layer is the canonical kernel-space reference for stackable filesystem architecture. Its design — a separation between mount-level operations (`struct vfsops`, e.g. `mount`, `unmount`, `root`, `statfs`) and per-vnode operations (`struct vop_vector`, e.g. `lookup`, `read`, `write`, `getattr`) — is the direct architectural ancestor of Grover's `GroverFileSystem._*_impl` pattern and the conceptual foundation for FSP's protocol-level mount routing. The FreeBSD codebase teaches VFS locking discipline (vnode interlock, mount lock), namecache invalidation strategy, stackable filesystem composability (nullfs, unionfs), and event notification (inotify). This memo extracts concrete implications for both Grover's implementation and FSP's extension surface.

## Navigation Guide

**Entry points**: The VFS layer lives in `/sys/kern/vfs_*.c` and `/sys/sys/{vnode,mount}.h`. Start with `vfs_mount.c` (mount/unmount machinery), `vfs_lookup.c` (path resolution via namecache), and `vfs_vnops.c` (vnode-level syscall dispatch). **Mount operations** are defined in `struct vfsops` (mount.h:828) and dispatch via macros like `VFS_MOUNT`, `VFS_ROOT`, `VFS_SYNC`. **Vnode operations** dispatch through `const struct vop_vector *v_op` (vnode.h:142) — each filesystem type registers its own vector. **Concrete filesystems** exemplify patterns: `tmpfs` (in-memory, inode-like tree structure), `nullfs` (passthrough/stacking), `unionfs` (layered read-write), `devfs` (synthesized device tree), FUSE (userspace bridge). **Supporting subsystems**: `vfs_cache.c` (the namecache, critical for performance), `vfs_acl.c` and `vfs_extattr.c` (metadata operations), `vfs_inotify.c` (event watches, new in FreeBSD 14). **Isolation/capability**: `capsicum` (capability-based file descriptor scoping) integrated throughout syscalls and path resolution.

## Core Concepts (What They Did Well)

**Separation of concerns via dual dispatch**. Mount-level ops (mount, unmount, getroot, statfs, vget) live in `vfsops`; vnode-level ops (lookup, read, write, getattr, setattr, mkdir, unlink, rename) live in `vop_vector`. This split lets you mount instances of the same filesystem type at different paths with independent state, while sharing the operation dispatch table. Grover mirrors this: `GroverFileSystem` methods are mount-instance state (mounts dict, permissions), while `_*_impl` methods are the operation dispatch table.

**Namecache as critical path**. The name-to-vnode translation (lookup) goes through a hash table cache first (`vfs_cache.c`). On cache hit, no storage I/O occurs — just vnode pointer fetch. On miss, the filesystem's `vop_lookup` is invoked, which can then populate the cache. This is why `vfs_lookup.c` spends so much logic on cache invalidation: stale entries corrupt correctness. Grover's routing layer (longest-prefix mount matching + path rebasing) is cheaper than FreeBSD's walk-every-component model, but the principle — cache path-to-mount mappings aggressively — applies.

**Vnode type enum as contract**. Every vnode has a `v_type` (VREG, VDIR, VBLK, VCHR, VLNK, VSOCK, VFIFO, etc.). This contracts what operations are legal: you can't `readdir()` a VREG; you can't `read()` a VDIR. Grover's `kind` field (`"file"`, `"directory"`, `"chunk"`, `"version"`, `"connection"`, `"api"`) plays the same role, but is richer (distinguishes Grover-specific types) and flatter (no device types, no sockets — those are not part of an agent's "everything is a file" abstraction).

**Stackable filesystems as composability**. Nullfs is a do-nothing layer; unionfs layers writable changes atop a read-only lower FS. Both use a key pattern: wrap a lower vnode in an upper vnode, dispatch operations to the upper layer first, then delegate to the lower layer. Failures in the upper layer don't poison the lower FS. This is composability. Grover's mount routing achieves composition differently (no actual vnode wrapping), but the design goal is the same: compose several VFSs without coupling their implementations.

**Extended attributes and ACLs as metadata without storage**. FreeBSD ACL/xattr ops go through vop_setacl, vop_getacl, vop_listextattr, vop_setextattr — the filesystem decides how to store them. Some use on-disk inodes; some (FUSE) delegate to userspace; some (devfs) have no storage at all. Grover's `PermissionMap` is a simpler model (no per-file mode bits, no inheritance rules), but the principle — decouple metadata schema from storage — is sound. A future xattr-like extension should route through a VFS-level hook, not through path-level object types.

**Event notification as vnode observer**. Inotify watches are stored per-vnode in `v_pollinfo->vpi_inotify` (a TAILQ_HEAD). When a filesystem modifies a vnode, it calls `VN_KNOTE(vp, hint, flags)` with a change hint (CREATE, DELETE, WRITE, ATTRIB, etc.). Watchers are notified without the filesystem knowing their identities. This decoupling is elegant. FSP should follow this pattern for change notifications: register a listener per path/vnode equivalent, dispatch hints through a central knote-like mechanism, let the protocol layer decide how to encode hints into wire messages.

## Anti-Patterns & Regrets

**Vnode locking is a footgun.** Every vnode has `v_lock` (interlock) and `v_vnlock` (lock manager). The locking protocol is context-dependent: some operations require exclusive lock; some allow shared lock. Deadlock is easy if you don't understand the lock order (always acquire parent before child). Three separate lock fields (v_lock for identity/refcount, v_interlock for per-vnode flags, v_vnlock for the actual data lock) create fragility. Grover avoids this by being single-threaded (async await, not preemptive threads), but FSP — if it ever uses threads — should document lock-order invariants aggressively.

**Namecache invalidation is error-prone.** When a file is renamed or deleted, the cache must be invalidated, but partial invalidation is easy to get wrong. For example, if `a/b/c` is renamed to `a/b/d`, you must invalidate the cache entry for `a/b/d` (if it existed), the entry `a/b/c`, and the parent link `a/b`. Miss any one, and a subsequent lookup hits a stale entry. FreeBSD has had bugs here. Grover's longest-prefix-match routing doesn't cache individual path components (paths are resolved per-operation), so this class of bug is unlikely — but if caching is ever added (e.g., mount-point cache), invalidation discipline will be critical.

**Stackable filesystems have tight coupling risk.** Nullfs and unionfs work by calling into their lower layer's vop_* functions directly. If the lower layer's interface changes, the upper layer breaks. FreeBSD solves this by keeping vop_vector stable (spares in the struct for ABI compat), but the mental model is fragile. A better pattern: upper layers should only depend on a minimal, stable interface (e.g., "give me the root vnode, then call read/write on it"). Grover's mount routing avoids direct coupling (each FS is independent), but if a layer ever needs to examine a lower FS's internals, decouple carefully.

**FUSE marshalling is verbose and error-prone.** The FUSE protocol requires serializing requests/responses to a buffer. The `fiov_*` functions manage variable-length buffers. MAXFIDSZ=16 bytes (for filesystem identifiers) is a hard limit that can be exceeded if not careful. Kernel FUSE (in-kernel FUSE daemon implementation) reduces copying, but the syscall ABI is still complex. FSP should define a simpler wire format (e.g., JSON or msgpack) to avoid this class of bug. If binary serialization is chosen, make it extensible (version tags, unknown-field skipping).

**ACL semantics diverged across systems.** FreeBSD's ACL model is inspired by POSIX.1e (withdrawn) but not a complete match. Different filesystems (UFS, ZFS) support different ACL models. This divergence is not FreeBSD's fault — it's a fundamental problem — but it means ACLs are never truly portable. Grover's simpler `PermissionMap` (prefix-based, not per-file mode bits) avoids this, but if rich ACLs ever become necessary, expect to choose a dialect and document it.

**Capability-based isolation (capsicum) is orthogonal to filesystems.** FreeBSD layered capabilities onto file descriptors via capsicum, which integrates with path resolution (you can't follow symlinks if CAP_SYMLINK is not held). But the filesystem itself doesn't know about capabilities — they're enforced at the VFS layer. FSP should do the same: capabilities are a protocol-level concept, not part of the filesystem ops API. If Grover ever uses capabilities, they should be enforced at the boundary (the router), not inside the backend implementations.

## Implications for VFS (Implementation)

**1. Finalize the `_*_impl` pattern as the vnode-operations table.** Every `GroverFileSystem` subclass should override a consistent set of `_*_impl` methods. Compile a formal list: `_stat_impl`, `_listdir_impl`, `_read_impl`, `_write_impl`, `_mkdir_impl`, `_rmdir_impl`, `_rename_impl`, `_unlink_impl`, `_setattr_impl`, `_getattr_impl`, `_readlink_impl` (if symlinks ever land), `_listxattr_impl`, `_setxattr_impl`. Document each's signature, error contract, and locking assumptions (Grover is single-threaded, but making this explicit helps future backends). Add a terminal base method that raises `NotImplementedError` so missing implementations are caught at definition time.

**2. Mount routing must handle longest-prefix-first and path rebasing correctly.** The current `_resolve_terminal` (base.py:147) is correct: iterate sorted mount paths longest-first, rebase the request path, recurse. Ensure mount paths are always single segments (no `/a/b` mounts), enforce this at `add_mount`, and test that nested mounts (mount A at `/data`, mount B at `/data/archives`) rebase paths correctly at each level.

**3. Namecache analogue: cache mount-to-filesystem mappings, not individual path components.** Grover's current model re-resolves every request path on every operation. For performance, add an optional mount-path cache: `{path_prefix: terminal_filesystem}`. Invalidate on `add_mount` and `remove_mount`. Keep entries simple (no LRU, no TTL) because mounts are infrequent. This trades memory for latency and is safe because mount operations are explicit.

**4. Directory-level permissions (PermissionMap) should prefix-match like mounts do.** The `PermissionMap` model (override permission at `/path/prefix`) is sound, but the implementation should use the same longest-prefix-match algorithm as mounts. This prevents accidentally granting read access at `/data` while denying it at `/data/secrets` if the rule order is wrong.

**5. Extended attributes and ACLs should route through dedicated `_*_impl` methods.** Don't embed xattr/ACL logic inside `_stat_impl`. Instead: `_setxattr_impl(path, namespace, name, value)`, `_getxattr_impl(path, namespace, name) -> value`, `_listxattr_impl(path, namespace) -> [names]`, `_setacl_impl(path, acl_type, acl)`, `_getacl_impl(path, acl_type) -> acl`. This keeps the interface clean and lets backends opt-in to rich metadata without bloating the core API. Provide sensible defaults (xattr/ACL return `NotImplementedError` or empty list) so simple backends work out of the box.

**6. Stacking and composition: design `StackingFileSystem` as a protocol, not a base class.** If unionfs-style layering is needed (upper RW, lower RO), don't bake it into `GroverFileSystem`. Instead: define a "stacking filesystem" as one that has a `lower_fs` attribute and implements `_*_impl` by consulting the lower FS first, then delegating. Nullfs would be `pass_through(self, method_name) -> lower_fs._method_impl(rebased_path)`. Unionfs would be `try upper_impl(path); if ENOENT, try lower_impl(path)`. This is a protocol, not an inheritance relation — keep VFSs loosely coupled.

**7. Error taxonomy must distinguish storage errors from permission/validation errors.** Grover's `GroverResult` with `error` field already separates these semantically. Ensure `_*_impl` methods raise exceptions that map cleanly to the five error classes (NotFoundError, WriteConflictError, ValidationError, GraphError, MountError). Add a docstring to each `_*_impl` saying which exceptions it can raise and when. This helps backends reason about error handling and tests can verify exception contracts.

**8. User-scoping and per-user result rebasing.** Grover can expose different views of the same VFS per user via `user_scoped=True`. Ensure result paths are rebased *after* the backend returns them (in the router, before returning to the caller). This is already done in `_group_candidates_by_terminal` (base.py:169), but document the invariant: terminal filesystems always operate on absolute paths; the router rebases them to paths relative to the mount point before returning.

## Implications for FSP (Protocol)

**1. Define mount and unmount as core protocol operations.** FSP currently has stubs for `mkconn` and `add_mount`. The protocol should define these as first-class, alongside `read`, `write`, `mkdir`. Include: mount source (path or backend name), mount target (path in the namespace), mount options (ro/rw flags, backend-specific config), and mount result (success or error codes). Follow the FreeBSD model: `struct mount` encapsulates instance state; `struct vfsops` is the dispatch table. In the wire protocol: mount requests specify the backend type (analogous to vfsops registration), and subsequent operations on the mount path automatically route to that backend.

**2. Namespace semantics: paths always refer to the complete mounted tree, not individual mounts.** When the client does `fsp.read("/data/file")` and `/data` is a mount point, FSP should transparently resolve to the backend mounted there. The client should never see mount boundaries. This is different from explicit `mount` system calls (which are rare) — once a mount is in place, it's invisible. Document this in the protocol spec: "Paths traverse mount points transparently. Use the mount/unmount operations to manage the namespace; all other operations work across mount boundaries."

**3. Capability descriptors as mount-level access tokens.** Instead of POSIX-style mode bits, FSP could issue capability objects (e.g., UUID tokens) when a mount is created. Subsequent operations on that mount require presenting the token. This sidesteps the ACL complexity and gives the server fine-grained control: issue a read-only token for `/data/archive`, a read-write token for `/data/working`. The protocol would be: `mount(source, target, options) -> {mount_id, capability_token}`. Subsequent ops include the token. This is lighter than full capsicum but leverages the same principle.

**4. Error taxonomy must map cleanly to POSIX errno + FSP semantic classes.** Define: `FileNotFound (ENOENT)`, `PermissionDenied (EACCES)`, `Exists (EEXIST)`, `NotDirectory (ENOTDIR)`, `IsDirectory (EISDIR)`, `InvalidArgument (EINVAL)`, `IOError (EIO)`, `ReadOnly (EROFS)`, `NoSpace (ENOSPC)`, `NameTooLong (ENAMETOOLONG)`, `TooManyLinks (EMLINK)`. Map Grover's five error classes to these. When FSP transmits an error to the client, include both the semantic class and the POSIX name (for compatibility with tools that expect errno), but the semantic class is authoritative.

**5. Event notifications: define a knote-like subscription mechanism.** FSP clients should be able to subscribe to changes at a path prefix: `watch(path, events=[CREATED, DELETED, MODIFIED, ATTRIB]) -> subscription_id`. When the server detects changes (via inotify or internal hooks), send notifications to all subscribers: `{subscription_id, path, event_type, timestamp, details}`. Do *not* require the client to poll or re-issue commands. Grover's backend should call a `notify(path, event_type, details)` hook (empty by default) after every state-changing operation, which FSP marshals into wire notifications. This keeps backends simple and lets clients see changes in real-time.

**6. Directory listing must return a predictable, resumable format.** `readdir` should return a stream with: `{entries: [{name, kind, size, mtime, ...}], eof: bool, cursor: token}`. Clients can use the cursor to resume listing, which is essential for large directories (don't send 1M entries in one response). Cursor should be opaque to the client — the server can encode `cursor = (mount_id, directory_inode, offset)` or similar. This mirrors FreeBSD's `getdirentries(2)` (which returns offset for resumption) rather than `readdir(3)` (which hides it in opaque state).

**7. Define symlink-like redirects as metadata, not path-level indirection.** If redirection is needed (e.g., `/alias` redirects to `/target`), don't implement symlinks (which add path-resolution complexity and deadlock risk). Instead: store a redirect record in metadata: `{kind: "redirect", target: "/target"}`. Clients follow it explicitly via another API call. This sidesteps symlink loops, permission checks on the target, and locking headaches. Grover already uses this model via `.connections/`; FSP should formalize it.

**8. Capability-based scoping: respect file descriptor limits.** If FSP ever opens "handles" to files (for streaming reads/writes), each handle should consume resources (file descriptor slots, memory). Define limits per client and per server: `max_open_handles_per_client`, `max_open_handles_system_wide`. When a limit is hit, return `TooManyOpenFiles` and require the client to close older handles. This prevents resource exhaustion and forces long-lived clients to pipeline or re-open frequently accessed files.

## Open Questions

1. **Statfs and quota reporting.** FreeBSD's `VFS_STATFS` and `VFS_QUOTACTL` let filesystems report block usage, inode counts, and per-user quotas. Grover's `GroverResult` doesn't include statfs-like metadata. Should `ls /` include a summary of mounted filesystems' usage? This is low-priority (agents don't rely on it), but including it might help with debugging.

2. **Symlink semantics: redirect metadata vs. path-level indirection.** Grover's `.connections/` design avoids symlink complexity, but the agent may expect `ls -l /alias` to show `-> /target`. Should FSP special-case redirect entries in listing output to look like symlinks? Or teach agents to query `.connections/` for redirection info? The former is user-friendly; the latter is simpler to implement.

3. **Transaction boundaries and multi-operation consistency.** FreeBSD filesystems don't expose transactions (writes are atomic per inode, but multi-file operations like rename are not atomic across two vnodes). Grover's SQL backends could offer transactions, but FSP's protocol doesn't currently support batching ops. Should FSP define `begin`, `commit`, `rollback` to let clients atomically rename-and-update-metadata? This adds complexity but is valuable for reliable multi-step operations.

4. **Audit and change-log events.** FreeBSD's audit subsystem logs filesystem operations system-wide. Grover could emit per-operation logs (who, what, when). Should FSP expose a `audit_log(path, since=timestamp) -> [events]` operation? Low-priority, but useful for debugging and compliance.

5. **Performance and cache coherence.** If Grover ever runs in a multi-process or distributed environment, cache coherence between clients becomes critical. Grover's current single-process model sidesteps this, but documenting the assumption helps prevent future surprises.
