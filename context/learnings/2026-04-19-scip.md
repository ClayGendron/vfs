# SCIP (Source Code Intelligence Protocol) — Influences for VFS and FSP

> Reference repo: `/Users/claygendron/Git/Repos/scip`

## Overview

SCIP is a language-agnostic Protobuf-based protocol for indexing source code to power code navigation (Go to definition, Find references, Find implementations). Rather than encode a full graph, SCIP uses a document-local model where metadata is colocated: each document carries its symbols and occurrences, permitting streaming reads and parallelizable incremental production. SCIP is intentionally a *transmission* format, not a storage format — it assumes a consumer will reindex the data into a queryable database. For VFS, the symbolic indexing model and occurrence-based addressing offer patterns for how to structure hierarchical, symbol-level object addressing. For FSP, the message model and URI-like symbol naming provide a template for typed protocol payloads and namespaced object identity.

## Navigation Guide

- **scip.proto** (880 lines): The core Protobuf schema. Entry points: `Index` (top-level container), `Document` (per-file metadata, occurrences, symbols), `Occurrence` (source range + symbol + role + diagnostics), `SymbolInformation` (metadata: documentation, kind, relationships), `Symbol` + `Descriptor` (the symbol string grammar and its structured form).
- **docs/DESIGN.md**: The motivating principles. Key insight: SCIP avoids integer IDs and full graph encoding in favor of string-based symbol identification and document-local colocality. Designed as a push format (producer sends documents to consumer), not a pull format.
- **docs/CLI.md**: Commands for working with SCIP indexes: `lint` (validation), `print` (human inspect), `snapshot` (golden test generation), `test` (validation against test files), `stats`, `expt-convert` (to SQLite).
- **cmd/scip/**: Go CLI implementation. Entry points for understanding how symbol resolution, document parsing, and streaming work in practice.
- **bindings/**: Language bindings in Go, Rust, TypeScript, Haskell. The Go/Rust bindings are rich with utility functions (symbol parsing, range encoding, graph building).

Ignore: language-specific indexer implementations (`scip-java`, `scip-typescript`, etc.) unless studying a particular language's AST traversal.

## Core Concepts (What They Did Well)

**1. Symbol as URI-like Identifier (Occurrence-centric addressing)**

SCIP lifts the symbol from internal compiler data (often just a name string) into a fully qualified, unambiguous, language-independent *string* identifier. The grammar `<scheme> <package> (<descriptor>)+` is designed to be parseable by any language and to serve as a stable reference across tools. This parallels VFS's `path` field in `Entry`: both serve as the durable identity for a code entity, independent of storage implementation. FSP inherits this idea in symbol URIs for protocol payloads.

**2. Document-Local Colocality (Streaming & Parallelism)**

Instead of encoding a global graph (as LSIF did), SCIP stores per-document symbol lists and occurrences. This allows:
- A producer (indexer) to write files independently, one at a time, with no merge phase.
- A consumer (like the `expt-convert` command) to stream-read one document at a time and insert into a database without loading all data into memory.
- Incremental indexing: re-index one file, append its updated document to the index, rest of the index is unchanged.

VFS already implements this: `VFSResult.entries` is a flat, document-local row shape returned from every operation. FSP compounds this: each MCP tool returns the same `FSPResult` envelope. The pattern is powerful and worth repeating.

**3. Occurrence with Rich Metadata (Syntactic + Semantic)**

Each `Occurrence` carries:
- `range`: source position (3–4 int32 array, compact and language-encoding-agnostic)
- `symbol`: the symbol string (optional; absence means "no semantic info, highlight only")
- `symbol_roles`: bitset (Definition, Import, ReadAccess, WriteAccess, Generated, Test, ForwardDefinition)
- `syntax_kind`: rendering hint (Comment, Keyword, Identifier, StringLiteral, etc.)
- `diagnostics`: inline compiler warnings/errors
- `override_documentation`: localized hover text (e.g., for generic types with concrete args)
- `enclosing_range`: parent AST node for breadcrumb/hierarchy navigation

This is a model of "co-locate all relevant metadata for one syntactic atom" — no follow-up queries needed to render an occurrence. VFS's `Detail` type hints at similar colocality; FSP could adopt this for search results and file contents.

**4. Symbol Kind Explosion (Language-Specific Taxonomy)**

The `Kind` enum has 86 entries (Class, Method, Function, Field, Property, StaticMethod, AbstractMethod, Trait, Protocol, TypeClass, Macro, Enum, EnumMember, Union, etc.). This fine-grained categorization lets UI clients render appropriate icons and enable language-specific refactoring hints without parsing the symbol string. VFS's single `kind` field (file, directory, chunk, version, connection, api) is much cruder but sufficient for filesystem semantics. FSP's kind space should remain filesystem-centric (file, dir, symlink-equivalent) unless augmented with SCIP metadata.

**5. Relationship as Dual-Direction Semantic Link**

`Relationship` entries explicitly encode implementation/reference/type-definition relationships with boolean flags. This permits a consumer to build bidirectional indexes (e.g., "find all subtypes of this class") without re-parsing the graph. The design acknowledges that many queries are dual (superclass ↔ subclass, caller ↔ callee) and records both directions as metadata rather than as computed indices.

**6. Diagnostic Co-location**

Rather than a separate error/warning stream, `Occurrence` can carry `Diagnostic[]` inline. A client rendering a definition or reference can show any associated compiler errors without a separate round-trip. VFS could adopt this for write/edit operations that have validation failures.

## Anti-Patterns & Regrets

**1. Integer IDs as Compression (LSIF's Mistake, Not SCIP's)**

LSIF encoded vertices as integers (vertex 42 = definition of symbol X) to reduce JSON verbosity. This caused:
- Off-by-one bugs in indexers rippled across entire repos (blast radius problem).
- Debugging required a symbol table context (can't read raw data).
- Integer IDs had to be carefully managed in concurrent indexing.

SCIP explicitly avoided this. Lesson for VFS/FSP: string-based identity (paths, symbol strings) is more debuggable and more resilient than surrogate keys, even if it costs bytes. Stick with paths as identity; surrogate keys are a DB optimization, not part of the wire format.

**2. Graph Encoding (Avoided by SCIP, Inherent in Grover's Graph)**

SCIP avoided trying to be both a data format *and* an efficient query engine. Full adjacency lists require loading all relationships into memory; SCIP defers this to the consumer's database. Grover's graph module (predecessors, successors, pagerank) crosses this boundary — it assumes a queryable graph backend. This is fine but creates a dependency: symbol navigation requires the graph to be hydrated. SCIP indexes themselves can answer "go to definition" (find the occurrence with `Definition` role) and "find references" (find all occurrences with `symbol == X`), both streaming operations. For FSP, consider whether graph operations should be opt-in (only available if a graph backend is mounted).

**3. Position Encoding Complexity (Necessary Evil)**

Documents specify `position_encoding`: UTF-8 byte offsets (Go/Rust/C++), UTF-16 code units (JVM/.NET/JS), or UTF-32 code points (Python). This is necessary because:
- Different languages index strings differently (Python's `str` is abstract code points; Go's `string` is bytes).
- A consumer (like an IDE) needs to map SCIP ranges to editor positions without re-encoding.

But it's a source of subtle bugs. Grover currently assumes UTF-8 (Python strings), which is fine for internal use but would complicate an SCIP-to-Grover bridge. Document this assumption clearly.

**4. Streaming Breaks Bidirectional Lookups**

SCIP is designed as a push format: producer writes documents sequentially; consumer reads and indexes. But "find all implementations of interface X" requires reading the *entire* index to gather all relationship records. The spec acknowledges this in non-goals: "ease of writing consumers" is *not* a goal. Grover's mount-aware search (glob, grep) compounds this — a query must fan out to all mounts, not stream a single index. This is fine; just document that SCIP is for transmitting new index data, not for online queries.

**5. Symbol String Escaping (Backquote Syntax is Clunky)**

Symbol identifiers use backticks to escape special characters: a method named `foo-bar()` becomes `` foo`-`bar() ``. This is necessary but makes symbol strings fragile to parsing. Go/Rust bindings include parser utilities, but a consumer in another language must implement the escape grammar. For FSP, prefer simpler identifier schemes (UUID, base62-encoded paths) over ad-hoc escape syntax if possible.

**6. Precomputed Staleness**

Once an index is written, it's frozen. Updates require re-indexing the affected files and re-writing the index file (or appending new documents if streaming). There's no "invalidation API" — if an indexed symbol's documentation changes, you must re-index. This is fine for CI/CD (index on every commit) but costly for interactive editor environments. Grover's versioning system (`/.versions/`) addresses this partially by recording per-file history, but the graph (relationships) doesn't version. For FSP, if mounting an index artifact, make this trade-off explicit: "indexes are immutable; refresh by re-mounting."

**7. Coverage Gaps (Language-Specific Semantics)**

Kind enums don't cover all language constructs (though comments in the enum invite PRs). Some indexers emit custom kinds. If you depend on SCIP for agent-facing "is this a method or a function?" decisions, you'll hit gaps for esoteric languages. Grover's `kind` field is similarly incomplete (no "trait," no "protocol"). Document the assumption: "kind is a hint; symbol string parsing is the ground truth."

## Implications for VFS (Implementation)

**1. Precomputed Symbol-Addressable Indexes as Artifacts**

SCIP indexes can be mounted in Grover as read-only data sources. Each symbol occurrence becomes an `Entry` in a `VFSResult`:

```python
# Hypothetical SCIP mount
result = await vfs.grep("MyClass", paths=["/"], kind="symbol")
# Returns: entries with path = document relative path,
#          metadata = { symbol, symbol_roles, kind, documentation }
```

Implement this as a `SCIPFileSystem(VirtualFileSystem)` backend that:
- Loads a `.scip` index at construction.
- Exposes `_glob_impl` and `_grep_impl` to search symbol names and documentation.
- Populates `Entry.kind` from `SymbolInformation.kind`.
- Makes `_read_impl` return symbol metadata (documentation, relationships) instead of file content.

Schema: single `scip_objects` table (or extend `vfs_objects`) with columns: `path` (document), `symbol` (string), `kind` (int), `documentation` (text), `relationships` (JSON).

**2. Colocated Metadata in `GroverResult` / `Entry`**

SCIP's `Occurrence` shows how to avoid query fragmentation. Adopt this pattern:

- `Entry.symbol`: optional symbol identifier (for symbol-level operations).
- `Entry.kind`: populated for all row types (file, symbol, relationship).
- `Entry.detail` or `Entry.metadata`: JSON blob with symbol-specific fields (documentation, signature, enclosing_symbol).

When a user asks "show me the definition of X," don't return a path + make them follow up with a read. Return the path + the metadata in one `Entry`.

**3. Symbol URI Addressing**

VFS currently uses filesystem paths (`/src/main.py`) as the primary identifier. Consider adding a secondary symbol-level addressing scheme:

```
/src/main.py#MyClass.method()
/src/main.py#local:variable_name
```

This allows fine-grained mount routing (e.g., a SCIP mount at `/symbols/` could intercept symbol queries). Implement `_resolve_terminal` to check for `#` in the path and dispatch to a symbol-aware backend.

**4. Relationships as Graph Operations**

Grover's graph module (predecessors, successors) mirrors SCIP relationships. Populate the graph from a SCIP index:

```python
# Load SCIP index
scip_fs = SCIPFileSystem(...)
await vfs.add_mount("/scip", scip_fs)

# Graph queries now work on symbol relationships
result = await vfs.successors(path="/scip/path.py#MyClass")
```

This requires a graph backend that understands symbol identity, not just file paths.

**5. Position Encoding in Write Operations**

VFS's write/edit operations don't currently track source positions (they operate on full file content). If integrating with SCIP, add:

- `Document.position_encoding` as a schema field on `vfs_objects`.
- When editing a file that's been indexed, validate that edits don't invalidate recorded occurrence ranges.
- If ranges shift, emit a warning or trigger a re-index.

This is defensive: ensure agent edits don't silently break indexed symbol locations.

## Implications for FSP (Protocol)

**1. Typed Message Payloads (Learn from `Occurrence`)**

FSP currently returns a uniform `FSPResult` with flat `Entry[]`. Consider sub-typing based on the operation:

```python
# Current
result = fs.read("/file.py")  # Returns FSPResult(entries=[Entry(...)])

# Enhanced
result = fs.symbol_search("MyClass")  # Returns SymbolSearchResult(occurrences=[Occurrence(...)])
```

The `Occurrence` message is a model here: it bundles range, symbol, roles, syntax hint, and diagnostics in one structure. FSP's `Entry` could specialize:

- `FileEntry`: for read/ls/glob (path, size, content_hash, kind).
- `SymbolEntry`: for symbol queries (symbol, kind, documentation, relationships).
- `DiagnosticEntry`: for linting (path, range, message, severity).

Each entry type still satisfies a common interface (path, kind), but clients can deserialize to the right subtype.

**2. Symbol URI in Wire Protocol**

If FSP adds symbol-level operations, define a URI scheme for symbols:

```
grover+symbol:///project/src/main.py#MyClass.method()
grover+symbol:///scip/path.py#local:variable_name
```

Use this in `Occurrence`-like payloads to enable clients to construct hyperlinks and navigate symbol graphs. MCP clients can then offer "Go to definition" by parsing the URI and invoking a corresponding tool.

**3. Relationship Payloads**

FSP could expose a `find_implementations` or `find_relationships` tool that returns `Relationship`-like payloads:

```python
result = fs.find_implementations("/scip/interface.py#Animal")
# Returns entries with:
#   entry.symbol = "Dog"
#   entry.metadata = { is_implementation: True, is_reference: False }
```

This maps directly to SCIP's `Relationship` message and enables agent navigation of type hierarchies.

**4. Diagnostics as a Native FSP Type**

`Occurrence.diagnostics[]` demonstrates co-locating errors with source positions. FSP could expose diagnostics from indexers natively:

```python
result = await fs.lint("/src/")
# Returns entries with:
#   entry.path = "/src/main.py"
#   entry.kind = "diagnostic"
#   entry.metadata = { line, col, message, severity, code, source }
```

Indexers (scip-java, scip-python, etc.) already emit diagnostics; expose them through FSP without forcing clients to parse a separate log.

**5. Index Artifacts as Capabilities**

SCIP indexes are pre-computed artifacts. In FSP, treat them as capabilities that mount dynamically:

```python
# FSP server loads a directory of SCIP indexes
fs.add_scip_mount("/indexes/py/", "scip-python/0.1.0")
fs.add_scip_mount("/indexes/ts/", "scip-typescript/0.5.1")

# Now clients can query symbols and graph across languages:
result = await fs.grep("config", kind="symbol")  # Searches all SCIP mounts
```

Document the index version and producer tool as mount metadata so clients know what semantics to expect.

**6. Streaming Indexes for Large Codebases**

FSP's current model is single-server-process. For large SCIP indexes (multi-GB), consider streaming:

- Server vends an MCP tool `stream_index(path)` that returns an async generator of documents.
- Client buffers documents and writes to a local SQLite store (like `scip expt-convert` does).
- Client queries the local store, reducing server load.

This trades roundtrips for bandwidth and offloads compute to the client. Useful for language servers and IDE integrations.

## Open Questions

1. **Symbol resolution across repos.** SCIP's `external_symbols` field allows cross-repo references, but the consumer must load multiple indexes. Should Grover/FSP expose a "multi-repo symbol search" capability, or leave that to the agent? If exposed, how does mounting interact with symbol shadowing?

2. **Incremental indexing and versioning.** SCIP indexes are static. Grover's versioning system records file history. Should SCIP-mounted data version track index updates? (E.g., symbol documentation changed in a newer index version; what does "go to definition" return?)

3. **Position encoding as a source of bugs.** If Grover integrates SCIP ranges, edits on indexed files will shift ranges. Should VFS auto-reindex on write, or document as a manual step? What's the agent experience if ranges are stale?

4. **Kind space explosion.** SCIP has 86 kinds; Grover has 6 (file, directory, chunk, version, connection, api). If mounting SCIP, do we need to expose all 86 kinds to agents, or is a reduced taxonomy (function, class, variable, type, etc.) sufficient? Document the mapping either way.

5. **Graph backend availability.** Grover's graph operations assume a populated graph. If a SCIP mount is the only backend, can we materialize graph operations (successors, ancestors) without a separate graph engine? Or should the SCIP mount return "graph not available" errors?

6. **Query scoping in multi-mount scenarios.** If symbol "MyClass" exists in two SCIP indexes mounted at `/scip/proj1` and `/scip/proj2`, which one wins in a `symbol_search` query? Should results include both with mount context? This is a namespace collision problem that deserves a clear design.

---

**Summary**: SCIP is a transmission format, not a storage format — but its design principles (string-based identity, document-local colocality, rich metadata per occurrence, language-agnostic symbol grammar) translate well to VFS backend architecture and FSP wire protocols. The anti-patterns section (integer IDs, full graph encoding, staleness) defines what *not* to copy. The open questions identify gaps where Grover/FSP diverge from SCIP's read-only, transmit-once assumptions.
