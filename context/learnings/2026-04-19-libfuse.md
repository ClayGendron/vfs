# libfuse — Influences for VFS and FSP

> Reference repo: `/Users/claygendron/Git/Repos/libfuse`

## Overview

libfuse (https://github.com/libfuse/libfuse) is a production filesystem-in-userspace framework: kernel module, userspace library, and utilities enabling non-root users to implement filesystem semantics in application code. It is the standard reference implementation for this architectural pattern across Linux, macOS, and BSD. VFS and FSP both inherit from FUSE's wire-protocol concepts and high-level operation dispatch patterns, making libfuse the canonical precedent for "everything is a file" at the systems level.

- **`doc/kernel.txt`:** Kernel-userspace boundary, mount semantics, cache invalidation, and the fork/daemon lifecycle
- **`example/hello.c` (71 lines):** Minimal high-level filesystem — single synthetic file

- **`example/passthrough.c` (500+ lines):** Real-world pattern — mirrors host filesystem; shows `readdir` offset tracking, cache invalidation, xattr, and `direct_io` tuning
- **Skip:** Architecture docs on DAX, CUSE (character device FS), io-uring integration, Solaris/BSD ports

## Core Concepts (What They Did Well)

**1. Operations-table vtable dispatch (FUSE high-level API).**
`struct fuse_operations` is a pure vtable: every operation is a `int (*func)(args...)` returning errno or 0. Terminal implementations must override only what they need; unimplemented handlers are NULL. This is the pattern VFS's `_*_impl` terminal methods mirror exactly. It forces clean separation: routers call `_route_*`, backends implement `_*_impl`.

**2. Inode numbers as opaque handles.**
The low-level API (`fuse_lowlevel.h`) uses `fuse_ino_t` (uint64) as the persistent, across-request identity of an inode. The filesystem can treat it as an opaque token or decode it. This is more flexible than path-based dispatch for systems with hard links, renames, and concurrent access — but comes at the cost of needing to track generation numbers to prevent ABA problems. VFS's `grover_objects` table with per-object `id` is the database equivalent.

**3. Handle-based open files (`fuse_file_info.fh`).**
When a file is opened, the filesystem stores arbitrary state (file descriptor, object ID, position, etc.) in `fh`. Subsequent reads, writes, flushes, and releases carry this handle back without path lookup. This decouples file identity (inode) from file state (position, mode). VFS does not use handles — each operation re-resolves the path and creates a fresh session — but FSP's eventual HTTP+keep-alive server would need to track handles.

**4. Readdir offset tracking for pagination.**
FUSE readdir has two modes: stateless (whole directory in one call) or offset-tracked (filler callback returns 1 when buffer full, next readdir() passes new offset). The offset is opaque to the kernel — purely for the filesystem's use. This pattern scales to directories with millions of entries. VFS `ls` does not currently paginate; FSP's HTTP transport will need to.

**5. Cache control flags at the per-file and per-filesystem level.**
`fuse_config` has entry_timeout, attr_timeout, negative_timeout; `fuse_file_info` has direct_io, keep_cache, cache_readdir, nonseekable. This granularity lets a filesystem say "this file is mmap-able and cacheable" vs. "this file is a pipe, don't cache". VFS's permission model is coarser (read vs. write), but should consider adding cache hints.

**6. Xattr as first-class surface.**
`setxattr`, `getxattr`, `listxattr` are not bolted-on; they're in the main operations table. FUSE treats them like files — a metadata plane parallel to content. Grover's `.connections/` namespace is a filesystem-level equivalent; FSP should expose xattr ops for agent-friendly metadata.

**7. Mode bits as the permission boundary.**
FUSE chmod/chown/umask complete the POSIX permission model. This is what VFS intentionally rejects (see 2026-04-18-posix-and-related-standards.md). Document this divergence clearly so agents don't expect `chmod` on a Grover path.

**8. Flush, fsync, release lifecycle.**
- `flush`: called on every fd close; used to write back dirty data from page cache
- `fsync`: explicit sync (fd must remain open)
- `release`: called when the last fd to a file is closed; good place to free handles
This sequence is critical for write ordering. VFS's "write before commit" rule (see 2026-04-18-swe-agent-aci-principles.md) is a simplification that avoids the flush/fsync tangle.

## Anti-Patterns & Regrets

**1. Stat storms on readdir.**
If a FUSE filesystem calls `stat()` for every entry in `readdir()`, the performance cliff is severe. The `FUSE_READDIR_PLUS` mode allows the filesystem to return attributes (inode number, mode, size) *within* readdir so the kernel can prefill its inode cache without separate lookups. Many implementations still miss this.
- *Implication for VFS:* `ls` should return kind + size in the entry, not require a follow-up stat for each entry. Currently, `_ls_impl` returns `Entry` with metadata; ensure this is efficient in database backends.

**2. Path-based dispatch race conditions.**
A file can be renamed between `lookup(path) → open(path)`. Low-level FUSE avoids this by returning an inode number from lookup, then passing the inode to all subsequent operations. High-level FUSE's path-based dispatch is vulnerable.
- *Implication for VFS:* Path-based routing is intentional for simplicity; document that concurrent renames can cause TOCTOU issues if an agent relies on path stability across operations. Use transactions or version checks if strong semantics are needed.

**3. Cache correctness under concurrent external mutations.**
If the backing filesystem (for passthrough.c) is modified by another process, the FUSE cache can become stale. The entry/attr/negative timeouts are the only lever. Setting timeouts to 0 disables caching entirely, killing performance.
- *Implication for VFS:* When used as a database-backed filesystem, there is no external mutator — only Grover writes. But if a backend can be modified by external queries, cache TTLs become critical. FSP should document this.

**4. Hard-to-debug error translation.**
FUSE error returns are C errno codes (ENOENT = 2, EACCES = 13, EPERM = 1, etc.). An implementation that returns wrong errno or no error at all will confuse the kernel and user space.
- *Implication for VFS:* The `_classify_error` function translates Grover exceptions to five semantic classes. This is better than errno, but the mapping is lossy. Document the translation table so agents understand what went wrong. FSP's MCP transport should keep semantic classes, not convert to HTTP status codes and back.

**5. `keep_cache` vs. `direct_io` semantics are subtle.**
- `direct_io=1` bypasses the kernel page cache; writes go directly to userspace. Good for mmapped files or sparse storage.
- `keep_cache=1` keeps cached data even after close. Good when the filesystem controls all mutations.
- Setting both is an error. Getting the combination wrong causes data loss or stale reads.
- *Implication for VFS:* Don't expose these flags to agents. VFS should decide them at backend initialization time based on write semantics.

**6. Symlinks without loop detection lead to ELOOP issues.**
FUSE will follow symlinks recursively up to a limit, then return ELOOP. Filesystems that create circular symlink chains cause hangs or errors that are painful to debug.
- *Implication for VFS:* No symlinks exist in Grover; the connection graph avoids this. Document it clearly so agents don't try to create link chains.

**7. Operations table explosion.**
Modern FUSE has >40 operations (getattr, chmod, chown, truncate, open, read, write, flush, release, fsync, fallocate, opendir, readdir, releasedir, fsyncdir, poll, ioctl, create, mkdir, rmdir, unlink, rmlink, rename, link, symlink, setxattr, getxattr, listxattr, removexattr, lseek, copy_file_range, etc.). Most filesystems implement only a subset. The api is large enough that it's easy to miss an important operation.
- *Implication for VFS/FSP:* Define a minimal operation set explicitly. VFS currently has ~25 ops (read, write, edit, ls, delete, mkdir, move, copy, glob, grep, stat, plus graph ops). FSP ships ~15 (CRUD + nav + search). Don't grow these lists without justification — each new op is a constraint on backends.

**8. Inode numbering schemes are fragile.**
If a filesystem doesn't handle inode reuse and generation numbers correctly, the kernel's inode cache will serve stale data. The `FORGET` request exists to tell the filesystem "the kernel is dropping this inode from cache", but timing issues can still occur.
- *Implication for VFS:* VFS's path-based routing avoids this. But if a backend adds handle support, ensure that handle → inode mappings are versioned or invalidated on mutation.

## Implications for VFS (Implementation)

1. **Readdir pagination and offset tracking.** Currently, `_ls_impl` returns all entries at once. For databases with millions of rows, this is a memory bomb. Design a pagination protocol early: pass an offset token (opaque to the agent), backend maintains cursor state, returns next batch + new offset. FSP's HTTP server will need this; VFS's async router should propagate it to backends.

2. **Per-file caching directives.** Add an optional `cache_hint` field to `Entry` or `Detail`. VFS backends can set `cache_hint=("keep_cache", ttl_seconds)` for immutable files. The router can translate this into FSP response headers or MCP metadata.

3. **Directory-level permission enforcement.** VFS already has `PermissionMap` with prefix-based overrides. Ensure that `check_writable` is called *before* any operation that would mutate — don't rely on the backend to enforce rules. The permission check is a guardrail that fails fast, not a trust boundary.

4. **Xattr surface for metadata.** VFS's `.connections/` namespace is effective but agent-opaque. Consider adding `fuse_getxattr`-style operations (with prefix, e.g., `vfs.user_id`, `vfs.version`) so agents can attach metadata without building filesystem paths. FSP should expose these as `{"op": "getxattr", "path": "...", "name": "..."}`.

5. **Flush and fsync semantics.** VFS write/edit are synchronous — no separate flush needed. But if a backend adds append-only or streaming writes, ensure writes are ordered and durable before returning success to the agent. Don't promise ordering you can't deliver.

6. **Handle-based operations for HTTP/MCP transport.** When FSP gains persistent connections, open files should return a handle (opaque token, e.g., base64(path+version)). Subsequent reads use `{"op": "read", "handle": "...", "offset": ..., "size": ...}` instead of path lookup. This amortizes path resolution across many small reads.

7. **Error translation remains lossy.** VFS's five-class hierarchy is appropriate for agents. Don't try to recover POSIX errno from exceptions — keep semantic classes (NotFoundError, WriteConflictError, ValidationError, etc.) all the way to the agent. FSP's wire protocol should transport these names, not HTTP status codes.

8. **Minimize the operation surface.** VFS's ~25 ops are justified. FSP's 15 ops are appropriate. Don't add `poll`, `ioctl`, `lseek`, `setlk`/`getlk` (POSIX locks), `fallocate`, or `copy_file_range` unless an agent genuinely needs them. Each operation is a contract.

## Implications for FSP (Protocol)

1. **Capability negotiation on FUSE_INIT — translate to MCP handshake.** FUSE_INIT lets the filesystem negotiate features with the kernel (async read, atomic O_TRUNC, no open support, etc.). FSP should have an `init` tool that lets the agent query which operations are available and which backends are mounted. Example: `{"op": "init"}` returns `{"caps": ["read", "write", "edit", "ls", "glob", "grep"], "backends": [{"mount": "/data", "type": "database"}]}`.

2. **Readdir offset pagination.** FSP `ls` should support an optional `offset` parameter that's opaque and returned with results. Schema: `{"op": "ls", "path": "...", "offset": "abc123"}` → `{"entries": [...], "offset": "def456", "has_more": true}`. Don't force the agent to manage large directory listings in memory.

3. **Handle-based read/write for streaming.** For large file transfers, return a file handle on `open`. Example: `{"op": "open", "path": "...", "flags": "r"} → {"handle": "xyz"}`. Then `{"op": "read", "handle": "xyz", "size": 4096, "offset": 0}` avoids re-resolving the path 1000 times. The handle is opaque and expires on `close` or timeout.

4. **Xattr surface as JSON metadata.** FSP should expose `getxattr`, `setxattr`, `listxattr` as first-class ops, not hidden in `stat`. Schema: `{"op": "setxattr", "path": "...", "name": "user.version_id", "value": "..."}`. Agents can attach arbitrary metadata without building path-based namespaces.

5. **Error codes as semantic names, not errno.** FSP should return error objects with `{"error": "NotFoundError", "message": "...", "details": {...}}`, not HTTP 404 or errno 2. This matches VFS's exception model and is more readable to agents.

6. **Cache-control metadata.** Responses to `stat`, `read`, and `readdir` should include `{"cache_control": {"ttl_seconds": 3600, "strategy": "keep_cache"}}` if the backend provides it. Agents and intermediate proxies can use this to avoid redundant queries.

7. **Statfs (capacity) operation.** Add `statfs` to FSP so agents can query available space before large writes. Schema: `{"op": "statfs", "path": "..."}` → `{"total_bytes": ..., "free_bytes": ..., "available_bytes": ...}`. Useful for quota enforcement.

8. **Minimum set: CRUD + nav + search + metadata + admin.** Don't ship `chmod`, `chown`, `truncate`, `poll`, `ioctl`, or `lseek`. If an agent asks, return "not implemented" with a pointer to the documentation explaining why.

## Open Questions

1. **Inode reuse and versioning in database backends.** If VFS is backed by a database that recycles row IDs, how do we prevent the kernel (or future FUSE-like consumers) from serving stale cache entries? Should VFS add a generation counter to each object? Should FSP clients cache aggressively or conservatively?

2. **Readdir offset encoding.** Should the offset be a sequential integer (3rd entry onwards), a timestamp, or an opaque blob (e.g., serialized cursor position from the database)? Different backends will have different costs. Document the encoding or require backends to return it opaquely.

3. **Cross-mount consistency.** When a path resolves across multiple mounts (e.g., `/data/archive` where `/data` is a mount), what happens if the source mount is removed or the target mount is swapped? FUSE remounts are atomic; VFS's mount table is in-memory and mutable. Define the semantics explicitly so agents don't encounter race conditions.

4. **Extended attributes vs. connection graph.** FUSE xattr is a flat namespace (`user.key`, `security.key`). VFS's `.connections/` is a path tree. FSP should support both and document when to use each. Is xattr for agent-supplied metadata and connections for agent-discovered relationships?

5. **Performance of path-based dispatch under high concurrency.** VFS routes by resolving the path and dispatching to the terminal filesystem. Does this scale to 10,000 concurrent requests? Should VFS or FSP implement request batching or connection pooling? Benchmark.

6. **Backwards compatibility with agents trained on POSIX semantics.** Agents assume `chmod` works, `.` and `..` exist in `ls`, symlinks can be created. VFS/FSP's deviations are intentional but need agent education. Should there be stub operations that return "not supported" vs. 404? The SWE-agent paper suggests error + explanation is better than silence.
